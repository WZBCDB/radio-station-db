[1] SQL INJECTION IN SEARCH FILTER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Location: src/app/dashboard/page.tsx, lines 31-33
Severity: CRITICAL
Exploitability: HIGH
Fix Complexity: LOW (5 minutes)

Problem:
  User search input directly interpolated into Supabase filter string:
  query.or(`title.ilike.%${filters.q}%,artist.ilike.%${filters.q}%`)

Attack:
  Search query: "test%,artist.eq.admin"
  Result: Returns admin's items + normal search results (data exposure)

Impact:
  - Information disclosure about other users' collections
  - Enumeration attacks possible
  - Privacy violation

Fix:
  Use proper query builder or escape filter strings
  Estimated time: 5 minutes

[2] RLS POLICY ALLOWS NON-ADMIN MEDIA UPDATES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Location: supabase/migrations/001_schema.sql, lines 80-85
Severity: CRITICAL
Exploitability: HIGH
Fix Complexity: MEDIUM (20 minutes)

Problem:
  RLS policy allows ALL authenticated users to update ANY media record:
  "create policy 'Authenticated users can update media' ... using (true) with check (true)"
  
  No ownership verification (created_by != auth.uid())
  No role-based access control

Attack (Any Authenticated User):
  await supabase.from("media").update({ condition: "poor" }).eq("id", "other-user-id");
  
  Non-admin successfully corrupts another user's library data

Impact:
  - Data corruption (condition, location, genres, notes)
  - Vandalism of shared collections
  - Privilege escalation (non-admin can modify any data)
  - Denial of service (make collection unusable)

Fix:
  Split into two policies:
  1. "Users can update only their own media" (created_by = auth.uid())
  2. "Admins can update any media" (role = 'admin')
  Estimated time: 20 minutes

[4] SECURITY DEFINER FUNCTION WITHOUT ACCESS CHECKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Location: supabase/migrations/004_admin_features.sql, lines 5-12
Severity: HIGH
Exploitability: MEDIUM
Fix Complexity: MEDIUM (15 minutes)

  Function runs with SECURITY DEFINER (superuser permissions)
  No input validation or access control
  Can be called by any authenticated user

  create function public.increment_view_count(row_id uuid) ...
  security definer ... update public.media set view_count = view_count + 1 ...

  Rapid-fire 100+ calls to increment view count for any media
  Artificially inflate view counts
  Manipulate "Most Viewed" rankings

  - View count statistics unreliable
  - Privilege escalation (function runs with elevated permissions)

  Change to SECURITY INVOKER and add access control checks
  Verify caller has access to the media record
  Estimated time: 15 minutes

  - src/app/api/admin/users/[id]/route.ts, line 52 (user deletion)

Severity: HIGH
Exploitability: HIGH (automatic)
Fix Complexity: MEDIUM (25 minutes for all 3)
